# CASPaxos + Verus: Hierarchical Invariant Verification Design

**Date**: 2026-04-25
**Crate**: `verus-paxos`

## Goal

Demonstrate how Verus enables hierarchical composition of low-level specifications and
invariants such that high-level safety properties of the CASPaxos protocol are formally
verified. The implementation is real executable Rust; the proofs are compile-time
obligations enforced by Verus.

**In scope**: Safety (agreement, validity, linearizability). Multiple concurrent proposers.
Arbitrary partial failures and fail-stop failure of a minority of acceptors.

**Out of scope**: Liveness, leader election, reconfiguration.

---

## Architecture

Source files map one-to-one with verification layers. Each file depends only on files
below it in the list. No layer reaches past its immediate neighbor.

```
src/
  primitives.rs        ŌåÉ Layer 1: Ballot, Versioned<S>, Quorum specs
  acceptor.rs          ŌåÉ Layer 2: AcceptorState invariants + exec code
  quorum.rs            ŌåÉ Layer 3: Quorum-level chosen-value proofs
  history.rs           ŌåÉ Layer 4: ChosenHistory monotonicity invariant
  linearizability.rs   ŌåÉ Layer 5: Linearizability capstone theorem
  network.rs           ŌåÉ Simulated network with failure injection (exec only)
  cluster.rs           ŌåÉ CASPaxosCluster orchestrator (exec only)
  main.rs              ŌåÉ Demo scenarios
```

Verus is the compiler wrapper (`verus src/lib.rs`). No runtime Cargo dependencies.
`spec` and `proof` content is erased at runtime; only `exec` code executes.

## Layer 1: Primitives (`primitives.rs`)

### Types

```
Ballot        = (round: u64, proposer_id: u64)   // lexicographic order
Versioned<S>  = { version: u64, uuid: Uuid, state: S }
NodeId        = u64
Quorum        = Set<NodeId>                       // ghost/spec only
```

`S` is an opaque generic type parameter. The protocol never inspects `state`; all
invariants are stated in terms of `version` and `uuid`. The `state` field is an opaque
payload.

`Uuid` is a `u128` in spec mode (no rand dependency); in exec mode it is a `[u8; 16]`
generated by `Uuid::new_v4()` passed in as a ghost parameter so the spec does not depend
on the random generator.

### Spec functions

- `spec fn ballot_lt(a: Ballot, b: Ballot) -> bool` ŌĆö strict lexicographic order.
  `proof fn ballot_lt_is_strict_total_order` establishes irreflexivity, asymmetry,
  transitivity.

- `spec fn ballot_le(a: Ballot, b: Ballot) -> bool` ŌĆö `ballot_lt(a,b) || a == b`.

- `spec fn apply_cas<S>(f: spec_fn(S) -> S, v: Versioned<S>, new_uuid: Uuid)
  -> Versioned<S>` ŌĆö returns `{ version: v.version + 1, uuid: new_uuid, state: f(v.state) }`.
  The only place `state` is touched in spec mode.

- `spec fn is_quorum(q: Quorum, cluster_size: u64) -> bool` ŌĆö `|q| * 2 > cluster_size`.

- `proof fn quorums_intersect(q1: Quorum, q2: Quorum, n: u64)` ŌĆö pigeonhole: any two
  quorums over a cluster of size `n` share at least one member.

### Failure model

No `is_live` predicate. Safety is unconditional ŌĆö it does not require any particular
failure rate. The failure model is captured implicitly by `participating_quorum`:

- `spec fn participating_quorum(round: RoundId) -> Quorum` ŌĆö the set of acceptors that
  actually responded within a given round's message light cone.

Safety invariants are stated as properties of completed rounds (those that have a
`participating_quorum`). Whether any round completes is a liveness question, not modeled
here. An acceptor absent from a round's participating quorum simply did not contribute to
that round's outcome, for any reason.

## Layer 2: Acceptor State Invariants (`acceptor.rs`)

### Type

```
AcceptorState<S> = {
    promised: Option<Ballot>,
    accepted: Option<(Ballot, Versioned<S>)>,
}
```

### Core invariant

```
INV_ACCEPTOR: if accepted == Some((b, v)) then
              exists p, promised == Some(p) && ballot_le(b, p)
```

An acceptor's accepted ballot never exceeds its promised ballot.

### Exec transitions

Both functions carry `requires INV_ACCEPTOR(state)` and `ensures INV_ACCEPTOR(result.state)`.

**`handle_prepare(state, b) -> (AcceptorState<S>, PrepareResponse<S>)`**
- If `ballot_lt(state.promised, Some(b))`: set `promised = Some(b)`, return
  `Promise(b, state.accepted)`.
- Otherwise: return `Nack`.
- Ensures: if `Promise` returned, `result.promised == Some(b)`.

**`handle_accept(state, b, v) -> (AcceptorState<S>, AcceptResponse)`**
- If `ballot_le(state.promised, Some(b))`: set `accepted = Some((b, v))`, return
  `Accepted(b)`.
- Otherwise: return `Nack`.
- Ensures: if `Accepted` returned, `result.accepted == Some((b, v))`.

### Proposer spec (spec-only)

`spec fn select_value<S>(responses: Seq<Promise<S>>, own_value: Versioned<S>)
-> Versioned<S>` ŌĆö returns the accepted value with the highest ballot among all Phase 1
responses, or `own_value` if no acceptor had accepted anything.

`proof fn select_value_is_highest` ŌĆö the returned value's ballot is `>=` all others in
the response set.

No `must_repair` obligation. Whether a proposer repairs a partially-replicated value
before applying its own CAS function is an implementation choice, not a spec requirement.
Repair is the natural strategy for reads (see Layer 4); writes may proceed directly.

## Layer 3: Quorum Invariants (`quorum.rs`)

### Chosen value definition

```
spec fn chosen<S>(
    acceptor_states: Map<NodeId, AcceptorState<S>>,
    b: Ballot,
    v: Versioned<S>,
    q: Quorum,
) -> bool {
    is_quorum(q, cluster_size)
    && forall |id| q.contains(id) ==>
        acceptor_states[id].accepted == Some((b, v))
}
```

### Lemmas

**`lemma_at_most_one_value_per_ballot`**
If `chosen(states, b, v1, q1)` and `chosen(states, b, v2, q2)` then `v1 == v2`.

*Proof*: `q1 Ōł® q2` is non-empty by `quorums_intersect`. The intersecting acceptor
accepted both `(b, v1)` and `(b, v2)`. `handle_accept` is deterministic and
`INV_ACCEPTOR` holds, so the acceptor holds exactly one accepted value per ballot.
Therefore `v1 == v2`.

**`lemma_higher_ballot_sees_chosen_version`**
If `chosen(states, b1, v1, q1)` and a proposer at ballot `b2 > b1` completed Phase 1
with quorum `q2`, then `select_value(phase1_responses(states, b2, q2)).version >= v1.version`.

*Proof*: `q1 Ōł® q2` non-empty; the intersecting acceptor accepted `(b1, v1)` and promised
`b2 >= b1`. `select_value` picks the highest accepted ballot, so it returns a version
`>= v1.version`. Proved by induction on ballot order; acts as the bridge to Layer 4.

**`lemma_version_unique_across_ballots`**
At most one `Versioned<S>` with any given `version` number can ever be chosen across all
ballots.

*Proof*: Induction on ballot order. Same-ballot case: `lemma_at_most_one_value_per_ballot`.
Cross-ballot case: `lemma_higher_ballot_sees_chosen_version` shows a later proposer
inherits the chosen version's causal lineage and cannot produce a conflicting value at the
same version number.

**`lemma_uuid_unique_among_chosen`**
If `chosen(states, b1, v1, q1)` and `chosen(states, b2, v2, q2)` and
`v1.version == v2.version` then `v1.uuid == v2.uuid`.

*Proof*: Immediate from `lemma_version_unique_across_ballots` which gives `v1 == v2`,
hence `v1.uuid == v2.uuid`.

**Note on transient acceptor state**: Different acceptors may transiently hold version `k`
with different UUIDs during concurrent proposals (competing proposers each reaching a
partial set of acceptors). The ballot discipline prevents more than one from reaching
quorum. The stale entry is overwritten when higher-ballot messages arrive. The
UUID-uniqueness invariant applies to chosen values only, not to transient acceptor state.
The same-ballot variant is trivially true: a single proposer issues exactly one value per
ballot.

## Layer 4: Chosen History Invariant (`history.rs`)

`ChosenHistory<S>` is a ghost-only `Seq<Versioned<S>>` that grows as rounds complete.
It exists only in `proof` mode; never in exec memory.

A companion ghost map `WitnessedValues<S>: Map<ProposerId, Versioned<S>>` records the
highest-versioned value each proposer observed in Phase 1 (committed or partially
replicated). This supplies the existential witness for `INV_CAUSAL_CHAIN` without
requiring intermediate versions to be committed.

### Invariants

**`INV_HISTORY_MONOTONE`**
`forall i < j, history[i].version < history[j].version`

Follows from `lemma_version_unique_across_ballots`: no two chosen values share a version,
and `apply_cas` strictly increments version by 1.

**`INV_CAUSAL_CHAIN`**
For each committed `v` at `history[i]`, there exists some `u` in `WitnessedValues`
with `u.version == v.version - 1` such that `v == apply_cas(f, u, v.uuid)` for some
`f: spec_fn(S) -> S`.

`u` may or may not be committed. A write proposer that finds a partially-replicated
value at version `k` in Phase 1 records it in `WitnessedValues` and applies its CAS
function to produce version `k+1` directly. Version `k` need not be committed first.
The invariant holds because `u` is the Phase 1 witness, not a committed history entry.

**`INV_READ_COMMITS_BEFORE_RETURNING`** (sits on the exec `read` function's `ensures`)
A read operation's return value `v` satisfies `chosen(acceptor_states, b, v, q)` for
some ballot `b` and quorum `q` at the time of return. A read may repair a
partially-replicated value it finds in Phase 1 (the natural implementation), or it may
discover that a higher version is already committed and return that instead. Either path
satisfies this ensures clause.

**`proof fn history_append_preserves_invariants`**
Adding a newly chosen `Versioned<S>` preserves `INV_HISTORY_MONOTONE` and
`INV_CAUSAL_CHAIN`. Two cases: (a) normal append ŌĆö fresh `apply_cas` result from the
proposer's own CAS function; (b) repair append ŌĆö a proposer commits a partially-replicated
value found in Phase 1 as-is, the `WitnessedValues` witness already exists from the
original proposer's Phase 1.

## Layer 5: Linearizability Capstone (`linearizability.rs`)

Pure `spec` and `proof`. No exec code.

### Abstract register

```
spec fn abstract_register<S>(history: ChosenHistory<S>) -> Versioned<S> {
    history.last()
}
```

### Linearizability statement

```
proof fn cas_paxos_is_linearizable<S>(
    history: ChosenHistory<S>,
    witnessed: WitnessedValues<S>,
    op_log: Seq<Operation<S>>,
)
requires
    INV_HISTORY_MONOTONE(history),
    INV_CAUSAL_CHAIN(history, witnessed),
    INV_READ_COMMITS_BEFORE_RETURNING(op_log, history),
    operations_respect_real_time_order(op_log),
ensures
    exists |serialization: Seq<nat>|
        is_valid_linearization(history, witnessed, op_log, serialization)
```

A valid linearization assigns each completed operation a position in the causal chain
(using `WitnessedValues` for write linearization points and `ChosenHistory` for read
linearization points) such that:

1. The linearization point falls within the operation's real-time interval.
2. Operations are ordered consistently with version order.
3. Each operation's return value matches the register state at its linearization point.

### Key lemmas

**`lemma_version_order_respects_real_time`**
If operation `op1` completes before `op2` begins, then `op1`'s committed version `<`
`op2`'s committed version. `op2`'s Phase 1 happens after `op1`'s Phase 2 completed, so
`op2` sees at least `op1`'s version via `select_value`.

**`lemma_return_value_matches_history`**
Every read that commits version `k` returns `history[k]`. Follows from
`INV_READ_COMMITS_BEFORE_RETURNING`. Every write that commits version `k+1` returns
`apply_cas(f, witnessed[proposer_id])` which is exactly `history[k+1]` by
`INV_CAUSAL_CHAIN`.

The linearization is constructed by assigning each write its version's position in the
causal chain and each read its committed history position. Both lemmas confirm this
assignment is valid.

## Executable Layer

### `network.rs` ŌĆö `SimNetwork<S>`

- `messages: Vec<Message<S>>` ŌĆö pending message queue.
- `drop_rate: f64` ŌĆö probability of silently dropping any message (partial failure).
- `failed_nodes: HashSet<NodeId>` ŌĆö fail-stop nodes that neither send nor receive.
- `deliver_one()` ŌĆö pops the next non-dropped message and routes to target acceptor.
- No ordering guarantees. No Verus annotations. The network is the adversary.

### `cluster.rs` ŌĆö `CASPaxosCluster<S>`

- Owns `Vec<AcceptorNode<S>>` and `SimNetwork<S>`.
- `run_proposal(proposer_id, f, init_value)` drives one full two-phase round:
  Phase 1 ŌåÆ collect promises ŌåÆ `select_value` ŌåÆ `apply_cas(f, v, Uuid::new_v4())`
  ŌåÆ Phase 2 ŌåÆ collect accepts ŌåÆ return `Ok(Versioned<S>)` or `Err(Contention)`.
- Multiple concurrent `run_proposal` calls simulate concurrent proposers.

### `main.rs` ŌĆö Demo scenarios

1. **Happy path**: 5 acceptors, 0 failures, 2 concurrent proposers. Assert version
   sequence strictly monotone and UUID-per-version consistent at end.

2. **Minority failure**: 5 acceptors, 2 fail-stopped. One proposer completes; a second
   starts mid-flight to exercise the read-repair path.

3. **Contention storm**: 5 acceptors, 5 concurrent proposers, 20% drop rate. Most rounds
   return `Err(Contention)` and retry with higher ballots. Assert final acceptor states
   satisfy `INV_HISTORY_MONOTONE` and `INV_UUID_UNIQUE_AMONG_CHOSEN` by exhaustive
   inspection.

## Testing Strategy

**Tier 1 ŌĆö Verus (compile-time)**: `verus src/lib.rs` must pass. Covers all invariants
across Layers 1ŌĆō5. Verification failure is a build failure.

**Tier 2 ŌĆö Unit tests (`cargo test`)**: Exec functions tested in isolation on concrete
inputs. Boundary conditions per transition function. Does not re-prove invariants.

**Tier 3 ŌĆö Demo assertions**: Post-condition checks on final cluster state in `main.rs`
scenarios. Runtime checks exercising the exec layer through the network adversary.

No mocking. `SimNetwork` is the actual adversary model, not a test double.
